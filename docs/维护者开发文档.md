# Deuterium Lottery 维护者开发文档

本文档面向未来维护本项目的人类开发者和 AI 代码代理。它不是玩家说明，也不是宣传文案，而是用于快速理解项目边界、生产安全要求、核心流程和后续迭代注意事项。

当前维护基线：

- 稳定生产分支：`main`
- 节日公益活动测试分支：`holiday-public-event-test`
- 当前测试版本：`1.1.1-SNAPSHOT`
- Minecraft / Bukkit API：`1.20.1`
- Java：`17`
- 经济接口：Vault，生产目标经济插件为 XConomy
- 数据库：SQLite，文件位于 `plugins/lottery/lottery.db`

## 最高优先级原则

生产环境数据安全永远高于功能速度。

任何后续修改都必须默认保护以下内容：

- 已存在的 `plugins/lottery/config.yml`
- 已存在的 `plugins/lottery/lottery.db`
- 当前 daily / weekly 期数
- 当前奖池和 rollover
- 玩家已经购买的彩票或投注
- 邮箱绑定
- 流水记录
- 已保存的中奖记录

除非用户明确要求高危修复，并且已经说明备份，否则禁止写会自动清空、重置、覆盖生产数据的逻辑。

```text

  command/LotteryCommand.java        /lottery 指令入口
  config/                            配置读取、配置迁移、配置模型
  economy/EconomyService.java        Vault/XConomy 资金操作封装
  mail/MailService.java              SMTP 邮件通知
  menu/                              Bukkit 箱子菜单
  model/                             数据模型和枚举
  placeholder/LotteryExpansion.java  PlaceholderAPI 变量
  service/LotteryService.java        核心业务逻辑
  storage/StorageService.java        SQLite 存储层
  util/Text.java                     颜色、金额、倒计时等文本工具
```

核心业务尽量集中在 `LotteryService`，数据库访问集中在 `StorageService`。不要在菜单或指令里直接拼复杂业务流程。

## 彩票类型

当前有三种类型：

- `daily`：每日彩票，传统一二三等奖模式。
- `weekly`：每周彩票，传统一二三等奖模式。
- `holiday`：节日公益活动，玩家选择建筑主题分池，开奖抽出一个分池，命中分池的玩家按投注金额占比分奖。

`daily` 和 `weekly` 共享 `tickets` 表、`awards` 表和传统开奖逻辑。  
`holiday` 使用独立的 `holiday_bets` 表，不复用 `tickets` 表。

## 数据库表职责

```text
periods
  每种彩票类型当前期数、下次开奖时间、rollover、上一期头奖玩家

tickets
  daily / weekly 玩家购票记录

ticket_refunds
  daily / weekly 管理员退款记录
  当前期被管理员退款的票不再计入奖池、注数和开奖候选

holiday_bets
  holiday 玩家投注记录

holiday_refunds
  holiday 玩家自助退款和管理员退款记录
  已退款投注不再计入分池、奖池、限购和开奖候选

awards
  daily / weekly 中奖记录

ledger
  资金流水和关键业务流水

emails
  玩家邮箱绑定
```

数据库升级只能使用：

- `CREATE TABLE IF NOT EXISTS`
- `CREATE INDEX IF NOT EXISTS`
- 必要时 `ALTER TABLE ADD COLUMN`，且必须允许旧数据正常存在

禁止普通升级流程删除旧表、重建旧表、清空数据或重置 `periods`。

## 配置迁移

生产服可能从 `1.0.1` 直接替换到新 jar。`1.0.1` 的配置没有 `config-version`，这必须被视为正常情况。

配置迁移器：`ConfigMigrator`

设计要求：

- 旧配置没有 `config-version` 时，只追加或补充版本号。
- 旧配置没有 `holiday` 时，只追加默认 `holiday` 段。
- 旧配置已有 `holiday` 但没有 `holiday.refund` 时，只插入退款小节。
- 不覆盖已有 `daily`、`weekly`、`settings`、`mail`、`messages`。
- 所有新配置项必须在 `ConfigManager` 中有代码默认值。

替换 jar 后，管理员不应该被要求删除旧配置文件。

## 资金模型

插件不应该凭空制造货币。

常规资金流：

```text
玩家付款
-> settings.system-account 指定的系统账户
-> 开奖派奖 / 奖池不足退款 / 管理员退款
-> 剩余部分作为公益收益留在系统账户
```

生产服曾使用 `lottery` 或 `tax/TaX` 作为系统账户。维护者不要假设账户名固定，必须从配置读取：

```yaml
settings:
  system-account: "lottery"
```

所有收款、退款、派奖、加奖池都应写入 `ledger`。如果经济插件调用失败，不能把业务当作成功继续推进。

## daily / weekly 流程

购票：

1. 检查期数。
2. 检查玩家限购。
3. 检查余额。
4. 从玩家转账到系统账户。
5. 写入 `tickets` 和 `PURCHASE` 流水。
6. 如果扣款后写库失败，尝试立刻退款。

开奖：

1. 读取当前期未退款 `tickets`。
2. 计算票面总额、派奖池、rollover。
3. 无票则广播本期不开奖并进入下一期。
4. 奖池不足则退款，退款失败不得推进期数。
5. 奖池足够则抽取一二三等奖。
6. 派奖失败不得推进期数。
7. 全部成功后推进下一期。

管理员退款：

- 命令：`/lottery refund <玩家> <daily|weekly> [期数]`
- 退款成功后写入 `ticket_refunds` 和 `ADMIN_REFUND` 流水。
- 当前期被退款的票会从奖池、注数和开奖候选中排除。
- 历史期退款只影响记录和资金，不会回滚已经发生的开奖结果。

## holiday 流程

默认配置：

```yaml
holiday:
  enabled: false
  max-bets-per-player: 5
  bet-amounts:
    - 100
    - 300
    - 600
  reward-pool-percent: 90
  house-pool-percent: 10
  refund:
    enabled: true
    lock-before-minutes: 60
  outcomes:
    redstone:
      chance-percent: 45
    obsidian:
      chance-percent: 45
    gold:
      chance-percent: 10
```

玩家规则：

- 每期只能选择一个分池。
- 选定后，其他分池在 UI 中锁定。
- 可以在同一个分池继续投注，直到达到 `max-bets-per-player`。
- 开奖前 `holiday.refund.lock-before-minutes` 分钟之前可以自助退款。
- 自助退款只退当前期节日活动投注，不发邮件。
- 自助退款成功后可重新选择分池。

开奖规则：

1. 读取当前期未退款 `holiday_bets`。
2. 按 `chance-percent` 抽中一个结果。
3. 如果命中分池无人投注，派奖池 rollover 到下一期。
4. 如果命中分池有人投注，按该分池内玩家投注金额占比分奖。
5. 系统只保留 `house-pool-percent`，不承担固定倍率赔付。

管理员退款：

- 命令：`/lottery refund <玩家> holiday [期数] [redstone|obsidian|gold|all]`
- 不受开奖前锁定时间限制。
- 可退历史期。
- 可指定分池。
- 退款成功后写入 `holiday_refunds` 和 `HOLIDAY_ADMIN_REFUND` 流水。

## 菜单设计

主菜单是 27 格箱子 UI：

- 左侧：每日彩票
- 中间：节日公益活动
- 右侧：每周彩票
- 下方：邮箱绑定提示/状态

节日二级菜单：

- 展示红石、黑曜石、金块分池。
- 每个分池展示可投注额度。
- 玩家选择一个分池后，其他分池显示锁定。
- 玩家有节日投注时显示退款按钮。
- 退款锁定期内退款按钮显示不可退。

菜单刷新由 `menu-refresh-ticks` 控制，默认实时性较高。不要在刷新中做昂贵操作或网络操作。

## 邮件系统

邮件只用于中奖通知和测试邮件。

当前明确不发送邮件的行为：

- 玩家节日活动自助退款。

维护要求：

- 邮件正文不要包含 Minecraft 颜色代码。
- 邮件中的奖项名和彩票类型应使用面向玩家的中文显示名。
- 邮件发送失败只写日志，不补发。

## PlaceholderAPI

`LotteryExpansion` 提供基础变量。新增变量时需要：

- 兼容 daily / weekly / holiday。
- 不在变量查询中执行写库或经济操作。
- 不做过重计算。
- 在文档中记录变量名。

```powershell

```

可部署 jar：

```text
target/lottery-<version>-SNAPSHOT.jar
```

不要部署：

```text
target/original-lottery-<version>-SNAPSHOT.jar
```

发布归档目录：

```text

```

```text

```

## 生产升级检查清单

从 1.0.1 生产服升级到 1.1.x 测试版前，必须提醒管理员备份：

```text
plugins/lottery/config.yml
plugins/lottery/lottery.db
plugins/lottery/lottery.db-wal
plugins/lottery/lottery.db-shm
```

升级预期：

- 只替换 jar 可以启动。
- 旧 `config.yml` 不需要删除。
- 旧 `lottery.db` 不需要删除。
- 当前 daily / weekly 已购票不应丢失。
- 当前 daily / weekly 期数不应变化。
- 当前奖池不应被清空。
- 新表会自动创建。

上线前建议在测试服模拟：

1. 使用生产 1.0.1 的配置和数据库副本。
2. 替换为新 jar。
3. 启动服务器。
4. 执行 `/lottery period daily` 和 `/lottery period weekly`。
5. 确认期数、奖池、票数符合升级前数据。
6. 测试购买、开奖预览、历史查询。
7. 测试管理员退款。
8. 再考虑生产服替换。

## 常见风险点

金融插件无法做到数据库和外部经济插件的真正跨系统事务。当前实现通过流水、退款表和失败阻断降低风险，但后续仍应继续加强幂等性。

重点风险：

- 经济转账成功后，数据库写入失败。
- 数据库写入成功后，经济转账失败。
- 派奖过程中服务器崩溃。
- 管理员重复执行退款。
- XConomy 系统账户不存在或无法扣款。

后续优化方向：

- 增加更明确的 `PENDING / SUCCESS / FAILED` 结算状态。
- 将派奖和退款设计成可重试的补偿队列。
- 对每一笔资金流增加唯一业务键，避免重复支付。
- 增加测试服专用审计命令，快速对比系统账户、奖池计算和流水。

## AI 接手规则

未来 AI 维护本项目时，必须先阅读：

1. `docs/维护者开发文档.md`
2. `docs/开发与生产安全规范.md`
3. `docs/节日公益活动退款与历史查询-1.1.1.md`
4. `src/main/java/cn/xiwanzi/lottery/service/LotteryService.java`
5. `src/main/java/cn/xiwanzi/lottery/storage/StorageService.java`

在修改前必须明确：

- 是否会影响生产服现有配置。
- 是否会影响 daily / weekly 正在进行的期数。
- 是否会影响已经购买的彩票。
- 是否会改变资金流向。
- 是否需要数据库迁移。
- 是否需要新增权限。
- 是否需要补充文档。

在提交前必须至少执行：

```powershell
.\.tools\apache-maven-3.9.9\bin\mvn.cmd "-Dmaven.repo.local=.m2repo" -DskipTests package
git diff --check
```

如果改动涉及资金流，最终回复中必须明确说明：

- 改了哪些资金入口。
- 改了哪些资金出口。
- 失败时如何处理。
- 是否影响旧数据。

## 不要做的事

- 不要为了生成新配置而删除旧 `config.yml`。
- 不要为了测试而清空 `lottery.db`。
- 不要把 `original-*` jar 当作部署包。
- 不要让玩家自助退款 daily / weekly。
- 不要把节日活动设计成服务器固定倍率赔付。
- 不要在公开文档中写出高危 UUID 白名单。
- 不要在菜单刷新或 PAPI 变量中做网络请求。
- 不要在没有备份说明的情况下做 reset 类操作。

## 当前建议状态

`1.1.1-SNAPSHOT` 仍建议视为测试版。它可以用于测试服验证节日公益活动、管理员退款和指定期数历史查询。合并到 `main` 或部署到生产服前，应完成一次带生产数据库副本的升级演练。
