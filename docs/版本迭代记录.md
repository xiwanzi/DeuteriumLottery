# 版本迭代记录

本文档记录 `lottery` 插件从立项到当前测试版的版本演进、功能范围和生产环境注意事项。

核心原则：已经运行在生产服上的旧版本配置、数据库、当前期数、当前奖池、玩家购票记录和流水记录必须优先保护。任何新功能都只能作为增量加入，不能破坏既有运行状态。

## 版本总览

| 版本 | 状态 | 说明 |
|---|---|---|
| 1.0.0-SNAPSHOT | 历史初版 | 完成基础彩票软件、菜单、开奖、邮件、经济对接 |
| 1.0.1-SNAPSHOT | 生产稳定版 | 修复金融安全问题，补齐重置、奖池加注、开源文档 |
| 1.1.0-SNAPSHOT | 测试版 | 新增节日公益活动入口和增量配置迁移，不建议直接覆盖生产主分支 |

## 1.0.0-SNAPSHOT

### 版本定位

1.0.0 是项目第一版可运行插件，目标是完成建筑彩票的主体玩法，使服务器可以开始测试每日彩票和每周彩票。

### 已实现功能

- Bukkit/Spigot 1.20.1 插件主体。
- 插件名称 `lottery`，作者 `xiwanzi`。
- `/lottery` 主命令和 `/lot` 别名。
- 每日彩票。
- 每周彩票。
- 箱子式 GUI 菜单。
- 管理员可为指定玩家打开菜单，方便接入 TRMenu 或服务器菜单。
- Vault 经济接口，兼容 XConomy。
- SQLite 本地数据库。
- 玩家每期限购。
- 可配置票价、最低开奖奖池、奖金池比例、奖项人数和奖金比例。
- 固定每日、固定每周、间隔开奖三种调度方式。
- 自动开奖。
- 手动开奖。
- 奖池不足退款。
- 无人购票跳过。
- 全服开奖广播。
- 邮件通知中奖玩家。
- 玩家邮箱绑定、查询、解绑。
- 管理员修改其他玩家邮箱。
- QQ 号自动识别为 QQ 邮箱。
- PlaceholderAPI 基础变量。

### 主要限制

- 初版金融流程仍有进一步加固空间。
- 部分失败流程需要更清晰的失败流水和阻断逻辑。
- 配置升级能力还不完整。

### 保留文件

归档位置：

```text
releases/1.0.0/
```

包含：

- `lottery-1.0.0-SNAPSHOT.jar`
- `lottery-1.0.0-SNAPSHOT-shaded.jar`
- `original-lottery-1.0.0-SNAPSHOT.jar`

实际部署应优先使用 `lottery-1.0.0-SNAPSHOT.jar`，不要使用 `original-*`。

## 1.0.1-SNAPSHOT

### 版本定位

1.0.1 是当前主分支对应的成熟生产版。该版本围绕金融安全、生产可维护性、开源准备和管理员工具进行了加固。

### 关键修复

#### 购票扣款后写票失败

处理方式：

- 购票记录和购票流水使用数据库事务写入。
- 如果玩家扣款后数据库写入失败，插件会尝试立即退款。

#### 开奖结算重复执行风险

处理方式：

- 中奖记录使用唯一约束。
- 派奖和退款前检查是否已有对应流水。
- 已处理过的退款或派奖不会重复执行。

#### 经济失败仍推进期数

处理方式：

- 退款失败时不推进下一期。
- 派奖失败时不推进下一期。
- 失败时记录失败流水。

#### 奖金池比例和抽成比例冲突

处理方式：

- 以 `reward-pool-percent` 为准。
- `house-pool-percent` 自动归一化为 `100 - reward-pool-percent`。
- 配置不一致时写日志提醒。

### 新增功能

- `/lottery pool add <daily|weekly> <金额>`：管理员给当前期增加额外奖池。
- `/lottery reset <daily|weekly> confirm`：高危重置当前彩票类型。
- 高危重置支持 UUID 白名单。
- 邮件模板改为更正式的中奖通知。
- 邮件中的彩票类型和奖项名去除 Minecraft 颜色代码。
- 奖项显示改为中文：一等奖、二等奖、三等奖。
- 无人购票和奖池不足分成不同广播。
- 所有默认通知前缀统一为“建筑彩票”。
- 默认配置更完整，带中文注释。
- README、MIT LICENSE、`.gitignore`。
- 开源仓库初始化。

### 生产使用说明

1.0.1 可作为生产稳定版使用。

生产环境升级到 1.0.1 时注意：

- 服务器已有 `plugins/lottery/config.yml` 不会被 jar 自动覆盖。
- 如需使用新配置项，应手动合并配置或备份后重新生成配置。
- 数据库文件仍为 `plugins/lottery/lottery.db`。
- 旧期数、旧票据、旧奖池不会因为替换 jar 自动清空。

### 保留文件

归档位置：

- `lottery-1.0.1-SNAPSHOT-shaded.jar`
- `original-lottery-1.0.1-SNAPSHOT.jar`

实际部署应优先使用 `lottery-1.0.1-SNAPSHOT.jar`，不要使用 `original-*`。

## 1.1.0-SNAPSHOT

### 版本定位

1.1.0 是测试版，用于验证节日公益活动相关增量能力。它不应直接替代生产主分支，除非完成测试并确认生产升级路径安全。

### 已实现内容

- 新增配置迭代版本号：

```yaml
config-version: 2
```

- 新增第三种彩票类型：

```text
holiday
```

- 默认显示名称改为合规表达：

```text
节日公益活动
```

- 主菜单中间增加节日公益活动入口。
- `daily`、`weekly`、`holiday` 三种彩票类型在命令中统一支持。
- 新增 PlaceholderAPI 变量：
  - `%lottery_holiday_pool%`
  - `%lottery_holiday_tickets%`
  - `%lottery_holiday_next_draw%`
  - `%lottery_holiday_last_first_winner%`
- 新增 `ConfigMigrator`，用于旧配置增量追加新配置。
- 新增节日公益活动玩法方案文档。
- 新增主分支稳定版记录文档。

### 增量配置迁移设计

1.1.0 的配置迁移必须遵循只追加、不覆盖原则。

旧服首次替换 1.1.0 jar 时：

- 如果没有 `config-version`，只在配置文件末尾追加 `config-version: 2`。
- 如果没有 `holiday`，只追加 `holiday` 默认配置。
- 如果旧 `menu` 段落里没有 `holiday`，只追加独立的 `holiday-menu` 兼容段落。
- 不重写原有 `daily`。
- 不重写原有 `weekly`。
- 不重写原有 `settings`。
- 不重写邮件配置。
- 不重写广播配置。
- 不重写管理员自定义菜单槽位。

### 生产安全设计

1.1.0 加入了以下保护：

- 旧配置不会整份覆盖。
- 数据库不会重建。
- 已有 `daily` / `weekly` 期数不会被清空。
- 已有 `daily` / `weekly` 奖池不会被清空。
- 已有玩家购票记录不会被删除。
- 首次迁移时只创建缺失的 `holiday` 期数，不主动重算已有 `daily` / `weekly` 的下次开奖时间。
- 节日活动菜单槽位如果与旧配置中的每日、每周、邮箱入口冲突，会自动选择安全空槽。

### 当前限制

当前 1.1.0 只是先完成了节日公益活动入口和增量迁移能力。真正的“玩家对玩家动态奖池竞猜”玩法仍处于方案阶段，见：

```text
docs/节日公益活动玩法方案.md
```

该方案建议使用建筑主题选项：

- 红石
- 黑曜石
- 金块

玩法原则：

- 玩家押选项，不是押自己中奖。
- 玩家之间形成奖池。
- 服务器只抽公益手续费。
- 猜中选项的玩家按下注金额比例瓜分可派奖池。
- 不凭空生成货币，避免通货膨胀。

## 分支策略

推荐仓库分支策略：

| 分支 | 用途 |
|---|---|
| `main` | 生产稳定版，当前应保持 1.0.1 成熟功能 |
| `holiday-public-event-test` | 节日公益活动测试分支 |

`main` 不应直接承载未充分测试的新玩法。

节日公益活动完成测试前，不应合并到 `main`。

## 版本文件归档策略

所有已构建版本应保留到：

```text
releases/<version>/
```

这样即使 `target/` 被清理，仍然能找到历史版本 jar。

注意：

- `target/` 是构建临时目录，不应作为版本归档来源。
- `releases/` 是人工归档目录，应随仓库保留。
- 每次新版本编译完成后，应复制 runnable jar 到对应版本目录。

## 升级建议

生产服升级前必须：

1. 备份 `plugins/lottery/config.yml`。
2. 备份 `plugins/lottery/lottery.db`。
3. 如果服务器运行中备份 SQLite，需要同时备份：
   - `lottery.db`
   - `lottery.db-wal`
   - `lottery.db-shm`
4. 先在测试服替换 jar。
5. 确认当前期数、奖池、票据、邮箱数据没有异常。
6. 再考虑生产服替换。

## 当前结论

- 1.0.1 是当前成熟生产版。
- 1.1.0 是节日公益活动测试版。
- 后续所有功能应优先保护生产环境旧配置和旧数据库。
- 任何会改动已有期数、已有奖池、已有票据、已有流水的行为，都必须单独评审，不允许作为普通自动升级执行。
