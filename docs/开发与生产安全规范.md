# 开发与生产安全规范

本文档是 `lottery` 插件后续开发的约束文档。所有新功能、修复、重构、配置调整和数据库变更都必须遵守本文档。

最高优先级原则：

> 已经运行在生产环境中的旧版本数据安全永远优先。任何后续修改和新增功能，都不能破坏旧配置、旧数据库、正在进行的彩票期数、当前奖池、玩家购票记录、中奖记录和流水记录。

## 1. 生产安全红线

以下行为默认禁止，除非用户明确要求，并且已经完成备份和风险说明。

### 1.1 禁止覆盖生产配置

禁止在插件启动、reload 或升级时整份覆盖：

```text
plugins/lottery/config.yml
```

允许的做法：

- 新安装时使用 `saveDefaultConfig()` 生成默认配置。
- 旧配置升级时只追加缺失的新配置段落。
- 新配置必须有合理默认值。
- 如果新配置缺失，代码必须能使用兜底默认值运行。

禁止的做法：

- 直接重写整个 `config.yml`。
- 自动删除管理员自定义配置。
- 自动修改已有 `daily`、`weekly` 配置。
- 自动修改系统账户配置。
- 自动修改 SMTP 配置。

### 1.2 禁止破坏生产数据库

禁止在普通升级中执行：

- 删除数据库文件。
- 删除 `tickets` 表。
- 删除 `ledger` 表。
- 删除 `awards` 表。
- 删除 `periods` 表。
- 删除 `emails` 表。
- 重置已有期数。
- 清空已有奖池。
- 清空玩家邮箱绑定。

允许的数据库升级方式：

- `CREATE TABLE IF NOT EXISTS`
- `CREATE INDEX IF NOT EXISTS`
- `ALTER TABLE ADD COLUMN`，且必须有默认值或允许为空。
- 新功能使用新表，不破坏旧表。

### 1.3 禁止自动重置当前期

正在进行的彩票期数属于生产状态数据。

禁止在自动升级时改动：

- 当前期号。
- 当前下次开奖时间。
- 当前 rollover。
- 当前已购票据。
- 当前奖池。
- 上期一等奖记录。

如果确实需要修复期数问题，必须提供显式管理员命令，并要求二次确认。

### 1.4 禁止凭空制造不可控货币

彩票系统应默认是玩家资金再分配系统，而不是系统发钱工具。

常规资金流：

```text
玩家付款
→ 彩票系统账户
→ 按配置派奖或退款
→ 抽成留作公益活动或赛事基金
```

新增玩法必须说明：

- 奖金来源。
- 是否会凭空生成货币。
- 是否可能造成通货膨胀。
- 手续费如何进入系统账户。

节日公益活动原则：

- 玩家之间形成奖池。
- 服务器只抽公益手续费。
- 不使用固定倍率让服务器作为庄家赔付。

## 2. 配置迭代规范

### 2.1 配置版本号

配置文件必须保留：

```yaml
config-version: 2
```

后续每次新增配置结构，应提升配置版本号。

版本号用途：

- 判断是否需要追加新配置。
- 帮助管理员知道当前配置结构版本。
- 记录升级行为。

注意：配置版本号不能作为覆盖配置的理由。

### 2.2 增量迁移

配置迁移器只能做追加操作。

推荐模式：

```text
如果缺少新顶级配置段落：
  在文件末尾追加新段落
否则：
  不修改已有内容
```

对旧配置兼容时，可以使用独立兼容段落。例如：

```yaml
holiday-menu:
  slot: 13
  material: "EMERALD"
  name: "&6&l节日公益活动"
```

这样可以避免插入到已有 `menu` 段落内部，减少 YAML 重写风险。

### 2.3 代码兜底

所有新配置都必须在代码里提供默认值。

如果生产环境旧配置没有新字段，插件仍应正常启动。

示例：

```java
config.getString("menu.holiday.name", "&6&l节日公益活动")
```

## 3. 数据库开发规范

### 3.1 表职责

当前数据库文件：

```text
plugins/lottery/lottery.db
```

当前表职责：

- `periods`：当前期数、开奖时间、滚入奖池、上期一等奖。
- `tickets`：玩家购票记录。
- `awards`：中奖记录。
- `ledger`：资金流水。
- `emails`：邮箱绑定。

### 3.2 资金流水必须可审计

涉及资金的行为必须写流水：

- PURCHASE
- REFUND
- REFUND_FAILED
- PRIZE
- PRIZE_FAILED
- HOUSE_RETAIN
- POOL_ADD
- RESET

新增玩法如果涉及资金，应新增清晰 action，例如：

- HOLIDAY_BET
- HOLIDAY_PAYOUT
- HOLIDAY_PAYOUT_FAILED
- HOLIDAY_FEE
- HOLIDAY_ROLLOVER

### 3.3 事务要求

以下操作必须使用事务或补偿逻辑：

- 写票据和写流水。
- 保存中奖结果和派奖。
- 重置类高危操作。
- 新增节日活动下注记录。

金融类插件不能出现“扣钱成功但记录失败且无补偿”的状态。

## 4. 功能开发规范

### 4.1 新玩法必须默认关闭或独立入口

实验性玩法不应影响已有玩家习惯。

推荐：

- 新玩法独立配置。
- 新玩法独立菜单入口。
- 新玩法可通过配置关闭。
- 新玩法不改变已有 daily/weekly 逻辑。

### 4.2 不要复用旧字段改变旧语义

禁止把旧字段改成新含义。

例如：

- `reward-pool-percent` 仍然表示进入奖金池比例。
- `min-total-pool` 仍然表示最低显示奖池。
- `max-purchases-per-player` 仍然表示每人每期限购。

如果新玩法需要新语义，应新增字段。

### 4.3 菜单改动要兼容旧槽位

新增菜单按钮时必须考虑旧配置可能已经自定义槽位。

要求：

- 新按钮默认槽位不能覆盖 daily、weekly、email。
- 如果配置冲突，代码应自动选择安全空槽。
- 不应强制改写旧菜单配置。

### 4.4 指令改动要兼容旧输入

新增彩票类型时：

- 旧指令仍应可用。
- 旧参数 `daily`、`weekly` 不变。
- 新参数作为增量加入。

禁止移除旧指令或改变旧权限含义。

## 5. 发布流程

### 5.1 分支规则

推荐：

```text
main = 生产稳定版
holiday-public-event-test = 节日公益活动测试版
```

未经测试的新功能不能直接提交到 `main`。

### 5.2 编译前检查

每次编译前必须自审：

1. 是否覆盖旧配置。
2. 是否删除旧数据库数据。
3. 是否改变旧命令行为。
4. 是否改变旧权限含义。
5. 是否改变旧奖池计算方式。
6. 是否改变旧开奖时间计算方式。
7. 是否会影响正在进行的 daily/weekly。
8. 是否有失败补偿或失败阻断。

### 5.3 编译命令

```powershell
mvn -DskipTests package
```

当前项目使用 Maven Shade。可部署 jar 通常为：

```text
target/lottery-<version>-SNAPSHOT.jar
```

不要部署：

```text
original-lottery-<version>-SNAPSHOT.jar
```

### 5.4 版本归档

每次成功构建后，应归档到：

```text
releases/<version>/
```

至少保留：

- runnable jar
- shaded jar
- original jar

## 6. 测试流程

### 6.1 本地测试

至少验证：

- 插件可启动。
- `/lottery` 可打开菜单。
- daily 可以购买。
- weekly 可以购买。
- holiday 入口不会覆盖旧入口。
- `/lottery reload` 不报错。
- 旧配置缺少新字段时仍可启动。

### 6.2 生产升级前测试

必须在测试服模拟：

1. 使用旧版 1.0.1 配置和数据库启动。
2. 替换为新 jar。
3. 检查配置是否只追加。
4. 检查 daily 当前期号不变。
5. 检查 weekly 当前期号不变。
6. 检查当前奖池不变。
7. 检查玩家票据不变。
8. 检查邮箱绑定不变。
9. 检查新玩法入口存在。
10. 检查旧玩法仍能开奖。

## 7. 回滚策略

如果新版本上线后出现问题：

1. 停服。
2. 备份当前 `plugins/lottery`。
3. 恢复上一个稳定 jar。
4. 如数据库未被破坏，可直接用原数据库启动。
5. 如果数据库结构已经新增字段或表，旧版本通常仍可忽略新表，但必须测试确认。
6. 如果出现资金异常，优先查看 `ledger` 表。

## 8. 节日公益活动开发边界

节日公益活动应保持健康合规命名：

- 使用“节日公益活动”。
- 使用建筑元素“红石、黑曜石、金块”。
- 避免在公开文案中使用直接赌博、赌场等强刺激词。

但玩法可以保留玩家对玩家的竞争性：

- 玩家押选项。
- 猜中者瓜分奖池。
- 热门选项预期收益低。
- 冷门选项风险高、收益高。
- 服务器只抽公益手续费。

## 9. 最终原则

开发时如果出现两个选择：

- 一个选择更快，但可能影响生产数据。
- 一个选择更保守，但需要多写兼容逻辑。

必须选择后者。

生产环境安全永远比功能速度重要。
