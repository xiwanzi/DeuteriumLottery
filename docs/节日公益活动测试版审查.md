# 节日公益活动测试版审查

分支：`holiday-public-event-test`

版本：`1.1.0-SNAPSHOT`

## 本次测试版范围

- 在主菜单中保留每日、每周彩票入口，并在中间加入“节日公益活动”入口。
- 节日公益活动默认关闭，配置项为 `holiday.enabled: false`。
- 节日玩法使用独立数据表 `holiday_bets`，不复用每日/每周的 `tickets` 表。
- 节日玩法采用玩家之间的分池竞争：
  - 玩家选择红石、黑曜石、金块之一参与。
  - 玩家选择配置内的投注额度。
  - 开奖抽出一个结果。
  - 命中结果的玩家按自己在该分池里的投注占比分配奖励池。
  - 如果无人命中，奖励池滚入下一期。
  - 服务器只保留 `house-pool-percent` 对应的公益手续费，不凭空增发货币。

## 生产安全检查

- 未修改每日彩票和每周彩票的购票表结构。
- 未删除 `tickets`、`awards`、`ledger`、`periods`、`emails` 旧表。
- 新增数据库对象只使用 `CREATE TABLE IF NOT EXISTS` 和 `CREATE INDEX IF NOT EXISTS`。
- 旧配置升级只追加缺失的 `config-version`、`holiday`、`holiday-menu`，不重写原配置文件。
- 如果生产环境已有旧版 `holiday` 配置但缺少新字段，代码会使用兜底默认值启动。
- `holiday.enabled: false` 时不会自动提醒或自动开奖，避免活动关闭时打扰服务器。
- 管理员仍可手动开奖，便于处理已有活动投注。

## 金融流程检查

- 玩家参与节日活动时，先从玩家账户转入系统账户，再写入 `holiday_bets` 和 `HOLIDAY_BET` 流水。
- 如果扣款后数据库写入失败，会立刻尝试从系统账户退回玩家。
- 节日开奖的结果会先写 `HOLIDAY_OUTCOME`，重试时复用同一个开奖结果，避免崩服后重新随机。
- 节日手续费写为 `HOLIDAY_FEE`。
- 节日派奖写为 `HOLIDAY_PAYOUT`。
- 节日派奖失败写为 `HOLIDAY_PAYOUT_FAILED`，并且不会推进到下一期。
- 节日退款失败写为 `REFUND_FAILED`，并且不会推进到下一期。
- 无人命中时写 `HOLIDAY_ROLLOVER`，奖励池滚入下一期。

## 业务流程检查

- 玩家打开 `/lottery` 后可看到每日、节日、每周三个入口。
- 节日活动关闭时，节日二级菜单显示活动未开放，玩家不能下注。
- 节日活动开启后，玩家进入二级菜单选择红石、黑曜石、金块和投注额度。
- 菜单展示当前奖池、该分池金额、个人投注次数、估算回报倍率。
- 管理员可用 `/lottery draw holiday` 手动开奖。
- 管理员可用 `/lottery preview holiday` 查看当前期状态。
- 管理员可用 `/lottery period holiday` 查看当前期。
- 管理员可用 `/lottery history holiday` 查看上一期节日活动结果。
- 管理员可用 `/lottery pool add holiday <金额>` 给当前节日活动奖池加注。

## 边界检查

- 活动关闭：不能下注，不自动开奖。
- 没有投注：不开奖，进入下一期，保留已有滚入奖池。
- 奖池不足：退款本期投注；退款失败则不推进期数。
- 无人命中：不退款，奖励池滚入下一期。
- 多人命中：按同分池投注金额占比分配。
- 同一玩家多次命中同一结果：按该玩家在命中分池的总投注金额合并派奖。
- 派奖中断后重启：已支付的玩家通过流水跳过，未支付玩家继续补发；全部成功后才推进期数。

## 暂不建议直接合并主分支的原因

该版本已经完成测试版开发和本地构建，但仍建议先在测试服验证：

- XConomy 对系统账户收付款是否稳定。
- SQLite 数据库升级后旧数据是否完全保留。
- 节日二级菜单在真实 Bukkit 1.20.1 混合端里显示是否符合预期。
- 开奖广播、历史查询、流水查询是否符合服务器运营习惯。
