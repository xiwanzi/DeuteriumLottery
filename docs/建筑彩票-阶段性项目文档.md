# 建筑彩票阶段性项目文档

版本记录日期：2026-04-27  
插件名称：lottery  
插件作者：xiwanzi  
当前版本：1.0.1-SNAPSHOT  
适配目标：Minecraft 1.20.1 Bukkit/Spigot 混合端  
当前产物：`target/lottery-1.0.1-SNAPSHOT.jar`

## 1. 项目定位

建筑彩票是为服务器设计的轻量级 Bukkit 彩票插件，核心目标是提供一个可配置、可热重载、金融流程相对可靠、适合中小型服务器日活规模的娱乐型经济玩法。

插件面向服务器玩家提供每日彩票和每周彩票两种彩票类型。玩家通过指令或服务器菜单入口购买彩票，开奖后系统自动派奖或退款，并可通过邮件通知中奖玩家。

本项目定位为公益性彩票玩法。彩票所得收益计划 100% 流入服务器建筑赛事的运营费用和奖金，用于支持后续建筑比赛和活动。

## 2. 已确认需求

### 2.1 基础需求

- 插件名称为 `lottery`。
- 作者为 `xiwanzi`。
- 服务器版本为 1.20.1 混合端。
- 插件需要轻量化。
- 支持插件自己的 reload 指令重载配置。
- 经济插件环境为 XConomy，通过 Vault 进行经济操作。
- 权限管理使用 LuckPerms。
- 变量环境支持 PlaceholderAPI。
- 支持 SQLite 本地存储。

### 2.2 彩票类型

插件包含两种彩票：

- 每日彩票：每日开奖。
- 每周彩票：每周开奖。

两种彩票均支持：

- 独立价格。
- 独立限购。
- 独立最低开奖奖池。
- 独立奖金池比例。
- 独立奖项人数和奖金比例。
- 独立开奖时间配置。

### 2.3 菜单需求

插件内置一个箱子式 GUI 菜单，用于玩家快速购买彩票。

菜单特点：

- 只能通过指令或外部菜单执行指令打开。
- 玩家可输入 `/lottery` 或 `/lot` 打开。
- 管理员或控制台可指定玩家打开，用于接入 TRMenu 或服务器菜单。
- 显示每日彩票和每周彩票入口。
- 显示当前开奖倒计时、价格、当前奖池、玩家本期已购注数、上期一等奖玩家。
- 菜单内容会定时刷新，奖池、倒计时、限购等实时更新。
- 菜单内包含邮箱绑定提醒。
- 未绑定时提示绑定 QQ 号或邮箱。
- 已绑定时显示当前绑定账号。

### 2.4 邮件通知需求

插件支持 SMTP 邮件通知。

已实现规则：

- 只有中奖玩家会收到邮件。
- 邮件发送失败不补发，只写日志。
- 玩家可绑定、查询、解绑自己的通知邮箱。
- 管理员可查询、修改、清除其他玩家的通知邮箱。
- 玩家绑定纯数字时，自动识别为 QQ 邮箱，即补全 `@qq.com`。
- 邮件正文不使用 Minecraft 颜色代码。
- 彩票类型和奖项名称在邮件中显示为纯文本中文。

### 2.5 开奖广播需求

开奖时全服广播：

- 广播前缀统一为“建筑彩票”。
- 显示第 N 期彩票开奖结果。
- 显示一等奖、二等奖、三等奖中奖玩家。
- 如果某个奖项因为人数不足没有中奖者，则隐藏对应奖项行。
- 如果无人购票，广播无人购票导致本期不开奖。
- 如果奖池不足，广播奖池不足并退款。

## 4. 数据与业务流程

### 4.1 购票流程

玩家购票流程：

1. 玩家打开彩票菜单。
2. 点击每日或每周彩票。
3. 插件检查本期个人限购。
4. 插件检查玩家余额。
5. 插件从玩家账户转账到系统彩票账户。
6. 插件写入票据记录和购票流水。
7. 菜单刷新玩家当前注数和奖池。

安全处理：

- 扣款后写票失败时，会尝试立即退款。
- 票据和购票流水在数据库中使用事务写入。

### 4.2 奖池计算

玩家购票金额 100% 会进入配置的系统账户，例如 `lottery` 或 `TaX`。

插件显示奖池和派发奖金池按配置比例计算：

```yaml
reward-pool-percent: 70
house-pool-percent: 30
```

实际以 `reward-pool-percent` 为准，`house-pool-percent` 会按 `100 - reward-pool-percent` 归一化。

例子：

```yaml
reward-pool-percent: 90
house-pool-percent: 10
```

代表每笔购票金额的 90% 进入显示奖池和派奖池，10% 留作收益。

### 4.3 开奖条件

`min-total-pool` 表示最低显示奖池，不是玩家购票总额。

例如：

```yaml
price: 100
reward-pool-percent: 90
min-total-pool: 1200
```

每卖 1 注，显示奖池增加：

```text
100 × 90% = 90
```

达到 `1200` 显示奖池才会开奖，大约需要：

```text
1200 / 90 = 13.34 注
```

也就是至少 14 注。

### 4.4 开奖流程

开奖流程：

1. 检查当前期是否到开奖时间，或管理员手动开奖。
2. 读取当前期所有票据。
3. 如果无人购票，本期跳过并进入下一期。
4. 如果显示奖池低于最低开奖奖池，则退款并进入下一期。
5. 如果满足开奖条件，计算奖金池。
6. 按奖项顺序抽取中奖者。
7. 从系统账户向中奖玩家发放奖金。
8. 写入中奖流水和中奖记录。
9. 邮件通知中奖玩家。
10. 全服广播开奖结果。
11. 推进到下一期。

### 4.5 中奖概率

当前插件采用按票抽取机制，不使用独立概率权重。

规则：

- 每 1 注彩票等于 1 个抽奖机会。
- 买 2 注的玩家，中奖基础机会约为买 1 注玩家的 2 倍。
- 同一期同种彩票里，默认同一玩家最多中一个奖。
- 开奖顺序为一等奖、二等奖、三等奖。

例子：

20 位玩家每人买 2 注，总票数为 40 注。任意一名玩家中一等奖的概率为：

```text
2 / 40 = 5%
```

### 4.6 奖金比例

奖项配置中的 `pool-percent` 表示奖金池占比，不是中奖概率。

例子：

```yaml
third:
  winners: 2
  pool-percent: 20
```

含义：

- 三等奖整体获得奖金池的 20%。
- 如果三等奖有 2 名中奖者，则 2 人平分这 20%。

## 6. 配置重点

### 6.1 系统账户

```yaml
settings:
  system-account: "lottery"
```

如果 XConomy 无法创建 `lottery` 账户，可以改为已有账户，例如：

```yaml
settings:
  system-account: "TaX"
```

当前流程：

- 玩家购买彩票：钱进入 `system-account`。
- 官方加奖：`/lottery pool add` 会向 `system-account` 增加金额，并计入当前显示奖池。
- 开奖派奖：从 `system-account` 扣钱发给玩家。
- 奖池不足退款：从 `system-account` 扣钱退给玩家。

如果使用 `TaX` 混账，插件显示奖池不会受账户原有税收余额影响，但实际派奖或退款要求 `TaX` 账户余额足够。

### 6.2 开奖时间

每种彩票支持三类开奖方式，且每种开奖方式都有独立开关：

```yaml
schedules:
  fixed-daily:
    enabled: true
    time: "21:00"
  fixed-weekly:
    enabled: false
    day: "SUNDAY"
    time: "21:00"
  interval:
    enabled: false
    minutes: 10
```

设计目的：

- 正式服可用固定每日或固定每周开奖。
- 测试阶段可用 interval，例如 10 分钟开奖一次。
- 同一种彩票同时启用多个开奖方式时，插件会暂停该彩票自动开奖并写日志提醒。

### 6.3 邮件配置

QQ 邮箱 SMTP 示例：

```yaml
mail:
  enabled: true
  smtp:
    host: "smtp.qq.com"
    port: 465
    ssl: true
    starttls: false
    username: "你的QQ号@qq.com"
    password: "你的SMTP授权码"
  sender:
    from: "你的QQ号@qq.com"
    name: "建筑彩票通知"
```

邮件模板支持变量：

- `%player%`
- `%lottery_type%`
- `%prize%`
- `%amount%`
- `%time%`

邮件中的彩票类型和奖项会被转成纯文本，不带 Minecraft 颜色代码。

## 7. 第一周期运营建议

### 7.1 第一周期活动比例

第一周建议：

```yaml
reward-pool-percent: 90
house-pool-percent: 10
```

说明：

- 第一周提高中奖体验。
- 长期不建议一直 90%，因为彩票账户留存较少。
- 第一周结束后可考虑改为 75% 或 80% 进入奖金池。

### 7.2 推荐价格与奖项配置

每日彩票建议：

```yaml
daily:
  price: 100
  max-purchases-per-player: 2
  min-total-pool: 1200
  reward-pool-percent: 90
  house-pool-percent: 10

  rewards:
    first:
      winners: 1
      pool-percent: 50
    second:
      winners: 1
      pool-percent: 25
    third:
      winners: 2
      pool-percent: 25
```

最低开奖时：

- 一等奖约 600，约 6 倍单注。
- 二等奖约 300，约 3 倍单注。
- 三等奖 2 人，每人约 150，约 1.5 倍单注。

每周彩票建议：

```yaml
weekly:
  price: 400
  max-purchases-per-player: 2
  min-total-pool: 7200
  reward-pool-percent: 90
  house-pool-percent: 10

  rewards:
    first:
      winners: 1
      pool-percent: 45
    second:
      winners: 2
      pool-percent: 30
    third:
      winners: 3
      pool-percent: 25
```

最低开奖时：

- 一等奖约 3240，约 8.1 倍单注。
- 二等奖 2 人，每人约 1080，约 2.7 倍单注。
- 三等奖 3 人，每人约 600，约 1.5 倍单注。

每周一等奖建议保持 1 人。原因是每周彩票频率低，玩家期待的是“本周大奖”，一等奖拆成 2 人会降低大奖感。

### 7.3 第一周官方加奖

已讨论方案：

- 每日彩票：每天官方加注 500 信用点。
- 每周彩票：第一周官方加注 3000 信用点。

说明：

- 官方额外加奖仅限第一周。
- 长期使用系统凭空加奖会增加服务器货币总量，可能造成通货膨胀。
- 如果官方加奖资金来自服务器已有账户，则只是资金转移，不会通胀。
- 如果通过 `/lottery pool add` 直接增加系统资金，则属于新增货币，适合短期活动，不适合长期常态化。

## 8. 通货膨胀说明

常规购票不会导致通货膨胀。

原因：

- 常规彩票资金完全来自玩家购票。
- 插件只是把玩家之间的资金重新分配。
- 一部分资金会留在系统账户，用于服务器公益活动或建筑赛事。

可能造成通胀的情况：

- 服务器通过系统命令直接生成新货币加入奖池。
- 长期官方补贴且补贴不是来自已有账户资金。

因此建议：

- 第一周可以做官方加奖活动。
- 后续常规期主要依靠玩家购票形成奖池。
- 彩票收益用于建筑赛事运营和奖金。

### 9.2 玩家 QA 精简稿

```text
Q：怎么购买彩票？
A：输入 /lottery 或 /lot，也可以在维度桥金融中打开彩票入口。

Q：中奖概率怎么算？
A：每买 1 注就是 1 次抽奖机会。比如 20 位玩家每人买 2 注，总共 40 注，你买 2 注，中一等奖概率就是 2/40 = 5%。

Q：开奖机制是什么？
A：插件自动随机开奖，先抽一等奖，再抽二等奖，最后抽三等奖。同一期同种彩票里，默认同一玩家最多中一个奖。

Q：如果奖池不够怎么办？
A：如果没有达到最低开奖奖池，本期不会开奖，系统会自动退款。

Q：中奖后怎么拿奖金？
A：不需要领取，开奖后奖金自动到账。

Q：可以离线中奖吗？
A：可以。只要买了彩票，即使开奖时不在线，也有机会中奖。

Q：怎么绑定 QQ 号或邮箱？
A：输入 /lottery bind QQ号或邮箱，或 /lot bind QQ号或邮箱。纯数字会自动识别为 QQ 邮箱。

Q：第一周为什么奖池更高？
A：第一周是试运行活动，服务器官方每日额外加奖 500 信用点，每周额外加奖 3000 信用点，且首周 90% 彩票资金进入奖金池。官方额外加奖仅限第一周，避免长期增加货币总量。

Q：建筑彩票会导致通货膨胀吗？
A：常规情况下不会，因为彩票资金完全来自玩家购票，不会凭空生成货币。

Q：彩票收益去哪了？
A：建筑彩票是公益性彩票，彩票所得收益会 100% 用于服务器建筑赛事运营和奖金。
```

## 10. 已处理安全审查项

### 10.1 购票扣款后写票失败

处理方式：

- 票据和购票流水用数据库事务写入。
- 如果扣款后写入失败，插件尝试退款。

### 10.2 开奖结算幂等风险

处理方式：

- 中奖记录使用唯一约束，避免同一玩家同一期同奖项重复记录。
- 派奖和退款前检查流水是否已存在。
- 如果派奖或退款失败，写失败流水，并阻止推进下一期。

### 10.3 经济失败仍推进期数

处理方式：

- 退款失败时不推进期数。
- 派奖失败时不推进期数。
- 成功退款或成功派奖后才推进。

### 10.4 奖金池比例和抽成比例

处理方式：

- 以 `reward-pool-percent` 为准。
- `house-pool-percent` 自动归一化为 `100 - reward-pool-percent`。
- 配置不一致时写日志提醒。

## 11. 当前已知注意事项

### 11.1 服务器已有配置不会自动覆盖

如果服务器已经生成过：

```text
plugins/lottery/config.yml
```

新版 jar 不会自动覆盖旧配置。需要手动合并新增配置，或备份后删除旧配置让插件重新生成。

### 11.2 `/lottery history` 当前无权限检查

当前 `/lottery history`、`/lottery history daily`、`/lottery history weekly` 没有权限检查。它只能查看上一期开奖结果，不涉及资金操作。

后续可考虑：

- 增加 `lottery.admin.history`。
- 或复用 `lottery.admin.preview`。
- 或保留给所有玩家查看，作为透明开奖记录。

### 11.3 `TaX` 混账注意事项

如果 `system-account` 设置为 `TaX`：

- 插件显示奖池不受 TaX 原余额影响。
- 实际派奖和退款会从 TaX 扣钱。
- 需要保证 TaX 余额足够。
- TaX 里原本的税收资金不会被插件单独区分。
